# 长期复利滤网

> 被 skill-optimizer 的阶段 3 和阶段 5 引用。
> 每个待修改项在应用前必须通过此滤网。

## 三问

修改前对每个发现回答三个问题。任一答案为"否"或"不确定"→ 降档或放弃。

### 滤网 1：系统性问题还是单次偶发？

```
判定标准：
  · 历史记忆中有同类记录？ → 系统性，维持或升档
  · 仅当前对话出现、历史无迹？ → 偶发，降一档
  · 无历史数据可查？ → 默认为偶发，保守降档
```

**反面案例：**
- 用户这次说"太快了跳过了"→ 给 skill 加确认步骤
- 但查历史：过去 20 次使用都没人抱怨过快
- 判定：单次偶发 → 降档，不建议改

### 滤网 2：三个月后回头看，庆幸还是后悔？

```
庆幸（推进）：
  · 改动让 skill 更通用、更清晰、更少误触发
  · 移除了特定场景的硬编码假设
  · 补充了遗漏的通用边界条件

后悔（放弃）：
  · 为今天的具体对话加了特殊分支
  · 把简单通用流程变成了复杂条件树
  · 加了一句"在某些情况下…"的模糊补充

不确定（降为 [中]）：
  · 进入对抗验证时专门问："这个修改的长期影响是什么？"
```

**反面案例：**
- skill 的步骤 3 在这次对话中没用上
- 建议："删掉步骤 3"
- 但步骤 3 是防止另一种常见错误的检查点
- 判定：删除会后悔 → 放弃

### 滤网 3：规则还是补丁？

```
规则（复利 ✓）：
  · 一条规则覆盖一类场景
  · 改完后，未来类似问题不再出现
  · 例子："所有触发词用 kebab-case" — 一条规则管所有 skill

补丁（✗ 禁止）：
  · 针对今天具体问题的特殊处理
  · 改完后只解决这一次的问题
  · 例子："当用户在下午用中文说'优化'时，额外检查 X" — 不可泛化
  · 除非用户明确声明"只是临时方案"
```

## 五条红线

以下修改类型**禁止执行**，即使置信度很高：

### 红线 1：为单一对话场景添加触发词

```
❌ 禁止：
  "这次用户说'帮我看看skill有没有问题'才触发的，
   加个触发词吧"
  
  问题：触发词表膨胀 → 误触发增加 → 每次都加载不需要的 skill
  正确做法：如果用户多次用不同说法表达同一意图，再考虑加。
           单次不算模式。
```

### 红线 2：为罕见的边缘路径增加步骤

```
❌ 禁止：
  "对话第 47 轮出现了 X 情况，skill 没覆盖到，
   加一个步骤处理 X"
  
  问题：X 是罕见情况，加步骤增加所有未来对话的认知负担
  正确做法：如果 X 是高频场景或安全关键场景，才加。
           否则留在边缘场景章节作为说明即可。
```

### 红线 3：删除"看起来没用"的步骤

```
❌ 禁止：
  "步骤 3 在这次对话中没用到，删掉"
  
  问题：可能防止了未观察到的问题
  正确做法：先 git log 查该步骤何时加入、当时解决什么问题。
           确认确实过时 → 可以删。
           无法确认 → 保留。
```

### 红线 4：把通用流程改成今天对话的形状

```
❌ 禁止：
  "今天对话的特殊之处在于 X，把流程改成先判断 X 再走分支"
  
  问题：过度拟合——今天对，明天错
  正确做法：今天的 X 如果是通用模式 → 提炼为通用规则。
           今天的 X 如果只是今天特殊 → 不改。
```

### 红线 5：未经对抗验证的 [中] 档修改

```
❌ 禁止：
  "置信度 65%，差不多可以改了"
  
  问题：65% 意味着 35% 概率改错——改错比不改更糟
  正确做法：[中] 档必须先过对抗验证，确认是 skill 问题而非上下文特殊。
           无法确认 → 保留 [中]，标注"需用户判断"。
```

## 判断框架：三个月后回头看

```
问自己：如果三个月后打开这个 skill 文件——

  看到这个改动，我会觉得：
    "当时改得好，现在还在受益"     → 做
    "当时为什么要加这个？现在已经不适用了" → 不做
    "这看起来像是临时的 workaround" → 不做

关键信号：
  · 改动移除了一个约束 → 通常是好的（更通用了）
  · 改动添加了一个条件判断 → 警惕（可能在走补丁路线）
  · 改动删除了一段内容 → 确认过 git log 再做
  · 改动替换了一段内容 → 新旧对比，新的是否明显更通用？
```
