# 置信度体系详解

> 被 skill-optimizer 的阶段 3 引用。

## 公式

```
每个发现 = 来源信号（基础分，可叠加） × 证据强度（0.4 / 0.7 / 1.0） → 原始分
原始分 + 影响范围修正（0% / -5% / -10%） → 最终置信度%
最终 cap 在 95%（永远留 5% 不确定性）
```

## 来源信号表

### 强信号（≥ 30 分）

| 信号 | 分值 | 含义 | 示例 |
|------|------|------|------|
| 用户手动纠正了 skill 行为 | +40 | 用户说"不对，应该这样……"并给出了不同的做法 | "不对，你应该先检查 X 再 Y" |
| debug-architect 🟢 输出 | +40 | 确定根因 + 高复利（≥15 复利分） | debug-architect 报告：根因是 skill X 缺少验证步骤 |
| 技能未被触发但本该触发 | +30 | 触发词表有对应条目但对话中未命中 | 用户说"检查一下"，但 skill X 的触发词是"审查"而非"检查" |

### 中等信号（15-29 分）

| 信号 | 分值 | 含义 | 示例 |
|------|------|------|------|
| 技能被触发但表现异常 | +25 | 触发了但执行了不该执行的步骤，或跳过了关键步骤 | skill 触发了但跳过了验证阶段直接执行 |
| debug-architect 🟡 输出 | +25 | 确定根因但低复利 | 特定环境的配置问题，别处不会重现 |
| 技能内容与对话需求矛盾 | +20 | 流程和实际需要不一致 | skill 要求先 A 后 B，但对话中 B 先做才合理 |
| debug-architect 🔵 输出 | +20 | 推测性根因，待验证 | "可能是 skill 的步骤顺序导致的，但不确定" |

### 弱信号（< 15 分）

| 信号 | 分值 | 含义 | 示例 |
|------|------|------|------|
| 历史记忆中有同类问题 | +15 | 跨会话重复出现 | 上周的对话中同一个 skill 也出过类似问题 |
| 纯静态审查发现 | +10 | 仅读 skill 文件就能发现的问题 | 正文中有矛盾的指令、冗余的步骤 |
| 工作流效率低 | +10 | 步骤冗余、来回过多、token 浪费 | "步骤 2 和步骤 4 做了同一件事" |
| debug-architect ⚪ 输出 | +5 | 存疑/一次性，几乎忽略 | 可能是用户操作失误导致的偶发错误 |

**叠加规则：** 多个信号可叠加（如：用户纠正 +40 + 历史同类 +15 = 55），相当于高起点进入下一阶段计算。

## 证据强度（乘数）

每次发现的证据等级由对话中可追溯的行为序列长度决定：

| 等级 | 乘数 | 判断条件 |
|------|------|---------|
| **强** | ×1.0 | 对话中有完整的"触发→执行→偏差→纠正"行为链，可以逐条引用 |
| **中** | ×0.7 | 需要推断几步但链条清晰（如："skill 触发了→但输出和预期不符→用户重新描述需求"） |
| **弱** | ×0.4 | 相关性存在但因果关系不确定（如："skill 触发后用户沉默了很久→可能是困惑也可能是思考"） |

**判断流程：**
```
能逐条引用对话中的具体行为吗？
  ├─ 能，且链条完整 → 强
  ├─ 能，但有几处需要推断 → 中
  └─ 只能指出相关性 → 弱
```

## 影响范围修正

| 范围 | 修正值 | 含义 | 判定方法 |
|------|--------|------|---------|
| **高频** | ±0% | 每次触发都受影响 | skill 的核心路径上的步骤 |
| **中频** | -5% | 特定条件下触发时受影响 | 分支路径、可选步骤 |
| **边缘** | -10% | 罕见场景 | 错误处理、特殊输入、极端参数 |

> 修正值直接加减在置信度百分比上，而非乘在分数上。例如：原始置信度 82%，边缘路径 -10% → 72%（降为 [中]）。

## 分级输出

| 档位 | 阈值 | 行为 |
|------|------|------|
| **[高]** | ≥80% | 强烈推荐修改，直接进入长期复利滤网 |
| **[中]** | 50-79% | 建议修改，先进对抗验证再进滤网 |
| **[低]** | <50% | 忽略，不出现在报告中 |

### 计算示例

**示例 1：强证据 + 强信号 → [高]**

```
发现：skill X 的步骤 3 在对话中被跳过
  来源信号：用户手动纠正 (+40) + 历史同类 (+15) = 55
  证据强度：强 ×1.0 → 55
  影响范围：高频 ±0% → 55%
  → [中] 档（仅靠用户纠正+历史还到不了高置信度）
```

**示例 2：双强信号 → [高]**

```
发现：skill Y 的触发词缺失
  来源信号：用户纠正 (+40) + debug-architect 🟢 (+40) = 80
  证据强度：强 ×1.0 → 80
  影响范围：高频 ±0% → 80%
  → [高] 档
```

**示例 3：弱信号 → [低]**

```
发现：skill Z 的步骤描述可以更清晰
  来源信号：静态审查 (+10)
  证据强度：中 ×0.7 → 7
  影响范围：中频 -5% → 2%
  → [低] 档，忽略
```

**示例 4：中等信号 + 边缘路径 → 降档**

```
发现：skill W 在某些极端输入时行为异常
  来源信号：表现异常 (+25) + 静态审查 (+10) = 35
  证据强度：中 ×0.7 → 24.5
  影响范围：边缘 -10% → 14.5%
  → [低] 档，忽略（边缘路径的修改风险大于收益）
```

## 对抗验证流程

仅对 [中] 档发现执行。

```
输入：[中] 档发现

1. 构造对抗问题：
   "这个发现可能不成立。
    请从另一个角度分析：有没有可能是上下文特殊而非 skill 缺陷？
    如果是 skill 问题，skill 的哪段具体文字导致了这个问题？
    如果是上下文特殊，为什么这次特殊？"

2. 交由一个新 agent 回答（不同于发现此问题的审查 agent）

3. 根据对抗结论升级/降级/维持：
   ├─ 确认是 skill 问题 → 升至 [高]，进入长期复利滤网
   ├─ 确认是上下文特殊 → 降为 [低]，附加说明（不出现在报告中）
   └─ 无法确认 → 保留 [中]，标注"需用户判断"，仍然展示
```

## 与 debug-architect 的置信度映射

当 skill-optimizer 在阶段 1.5 成功调用 debug-architect 并获得结果时：

```
debug-architect 输出           →  skill-optimizer 信号加分    →  说明
🟢 确定·高价值 (复利分 ≥15)     →  +40                        →  等同于用户手动纠正的强度
🟡 确定·低价值 (复利分 <15)     →  +25                        →  根因确定但影响范围小
🔵 推测·高价值                  →  +20                        →  需要额外验证才能确认
⚪ 存疑/一次性                  →  +5                         →  几乎忽略，仅在多信号叠加时有意义
```

**映射后的处理：**
- 映射后的信号加分只是来源信号的一部分，继续与其他信号叠加
- 叠加后的总分走正常的证据强度 × 影响范围 → 三档分级
- 不因 debug-architect 的评级而跳过 skill-optimizer 自己的滤网

**注意：** debug-architect 和 skill-optimizer 的 🟢🟡 含义不同：
- debug-architect 的 🟡 = 确定·低价值（仍然确定，只是影响小）
- skill-optimizer 的 [中] = 置信度 50-79%（不确定）
- skill-optimizer 报告中使用文字标注 [高]/[中]/[低] 以避免混淆
